python ctypes库pointer_type_cache内存泄漏场景和规避方案

发表于 2022-05-12 更新于 2022-05-22

简述

在使用Python过程中如果要调用ctypes.pointer(inst)来生成一个结构体或者一个类的指针并且传递给C接口的时候，如果inst的类型之前没有生成过pointer对象，那么ctypes在实例化pointer对象之后会生成一个{类型：指针类型}的键值对存储到ctypes._pointer_type_cache中，如果Inst的类型一直在变的话，这个缓存可能会不停的存键值对引发超大的内存占用。

先说说前提和应用场景

在公司中的一些跨部门合作中产生了这样一个使用场景

  1. 云端对接很多服务使用的是Go完成的代码，并且在代码中使用协程相关的特性
  2. 有一些加密代码是使用C写的，并且使用到了一些C写的系统调用和古董代码和加密代码，使用了一些posix系统调用(比如systemv标准的shm共享内存方案)
  3. 需要将部署在客户内网中的设备中的数据库中的数据同步一部分到云端，但是这部分数据库相关的接口只暴露了python相关的业务接口，所以主体的业务代码逻辑需要使用Python编码完成

之前就有人用C代码开发了一个中间件SDK来衔接不同业务代码之间的交互，所以我们实际在使用中，需要通过C接口来实现数据交互

根因和优化

根因，factory函数中定义的inner class在每次函数被调用的时候等价于声明了一个新的类，这个类实例在生成ctypes.pointer的时候会写新的pointer_type_cache，ctypes库可以通过这个cache来生成对应的指针类型，而不用每次来一个相同的类都声明新的指针类型，并且可以以此判断两个ctypes Pointer是否是相同的类型。显然，在现在的调用代码中，这个cache对于业务是没有影响的。

那么优化或者规避掉这种类似内存泄漏的场景的思路就可以从两个方面入手

  • 如果业务中的struct是可重复使用的，那么在使用过程中需要避免重复声明新的类

  • 如果业务中poitner_type_cache对于真实业务没有影响，那么可以清除掉这个缓存，用来避免对应的内存消耗

优化一：避免重复声明struct类

这个在实现上相对简单一些，由于写入共享内存的过程可以看作是一次深复制，所以传给SDK接口的struct地址可以是一个保有量，不用不停的根据数据长度动态声明

免责声明 代码仅供思路参考

当且仅当数据长度大于预设的struct变长数组长度的时候才声明新的类，这样可以避免不停的调用factory声明新的类把ctypes pointer type cache写爆

It may not be the only issue, but for sure the caching of each _Argtype: LP__Argtype pair in the dict _ctypes._pointer_type_cache is not insignificant. Memory usage should go down if you clear the cache.

The pointer and function type caches can be cleared with ctypes._reset_cache().

为了验证这个说法的有效性，再次执行mem_leak_demo死循环，并且用prasite-shell attach python进程，查看全局变量的长度

确实是在不停的变长，所以可以通过ctypes._reset_cache清空掉这个cache

If resetting the cache solves your problem, maybe that’s good enough. But generally the pointer cache is both necessary and beneficial, so personally I’d look for another way. For example, as far as I can see there’s no reason to customize _Argtype for each image. You could just define row as a double ** initialized to the array of pointers.

对于我们的业务来说，暂时用不到这个pointer_type缓存，所以定期的清理掉这个缓存对于节省进程的内存占用是有益的
